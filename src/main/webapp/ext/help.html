1 Introduction

The cREST Client extension is an http client with RESTful services in mind. The UI is divided into three high level sections including
  • Request Builder for constructing http requests
  • Responses for viewing/managing http responses
  • Request Store for viewing/editing your stored request data. This data includes saved requests, URI history, and header history.
Please feel free to speak up in http://groups.google.com/group/crest-client if you have questions/comments about the extension. Also let me know if there's anything I can do to make this help page more useful.

2 Request Builder

The request builder contains all the ui components you need in order to submit an http request. This is where you can specify a URI, select your http method, add any headers, or include a request entity for PUT/POST requests. The remainder of section 2 will describe some of the more notable components in the Request Builder. Let me know if you'd like anything else covered.

2.1 URI/Header Auto-complete

The autocomplete fields for URIs and headers will suggest options that it finds in your URI history, header history, or your saved requests. The URI and header history both get populated each time you use a new URI or header in a successful request. In addition, you can manually edit the URI/header history by clicking the pencil button next to the coorisponding autocomplete field. You can also get to your URI/header history from the request store button discussed below.

Saved requests will be covered in more detail TODO: below. Know that if you select one from the URI autocomplete dropdown list, it will populate the request builder with all of its data. This includes URI, method, and any headers or request entity that may be present. If you select a saved request from the header autocomplete dropdown, it will load all of that request's headers. Saved requests are named by you, so you should be able to easily distinguish them from your URIs and headers in the autocomplete suggestions.

Items for the autocomplete dropdown are selected in a case insensitive way. The suggested items only need to contain the term you've typed, not start with it. In addition, you can specify multiple terms by separating them with a space. This comes in handy when you have many URIs that only differ by a little.

2.1 Saved Requests

cREST lets you save requests for re-use later. Each response has a save button (disk icon) that allows you to save the request that produced the response. When clicked, the request editor will be launched so you can make some changes (typically giving it a better name), and then save the request.

Once the request is saved, it can be used to populate fields within the Request Builder. As previously mentioned, the names of your saved requests will show up in the URI and header autocomplete suggestions and when selected will populate corresponding Request Builder fields. There is also a new home button on the Request Builder tool bar. You can associate one of your saved request with it so that when it's clicked, the request will be loaded. Specifying this request is done with the "request store" covered in the next section.

Note that these requests don't necessarily need to represent a valid http request. You might create one that only has commonly used headers with no URI or request method.

2.1 Request Store

You can access/edit your stored data by clicking the folder icon. Three kinds of data are saved, each with their own tab:

  1. Saved Requests

    This tab allows you to manage your saved requests. The first row indicates which request is associated with your home button. By default, cREST associates a request with no URI, headers, or entity and a GET request method. Next you'll see your saved requests if you have any. They are ordered alphabetically. Each request has the following buttons (from left to right):

    1. Play - The saved request will be loaded into the Request Builder and a request will be submitted.
    2. Up Array - This loads the request in the Request Builder. Double clicking the request outside of the buttons also does this.
    3. Home - Sets the request to your home button.
    4. Copy - This allows you to copy the request. It will display the request editor with a new name which can be modified before you save.
    5. Edit - This allows you to edit the request.
    6. Trash - deletes the saved request.

  2. URI History

    These are all the URIs you've targeted that have returned successful responses. This tab displays a textarea for you to manually edit your URIs. These URIs are used for the URI autocomplete suggestions.

  3. Header History

    This tab is identical to the URI history, only it's for your headers and the header autocomplete.
